---

## 📖 Общая структура ответа

Когда препод спрашивает **"Что поставил и почему?"**, говори по схеме:

1. **Название программы** + **для какой роли** (из твоего билета)
2. **Какие задачи решает** (конкретика)
3. **Почему выбрал это, а не альтернативу** (сравнение)
4. **Какие ещё есть варианты** (покажи кругозор)

## 3. 🗄️ СУБД И БАЗЫ ДАННЫХ

### PostgreSQL
**Для билетов:** 1, 5, 8, 16, 18, 27, 28

**Что сказать:**
> PostgreSQL — самая мощная open-source СУБД. Поддерживает ACID, транзакции, JSON, PostGIS (геоданные). Для веба, финансов, гео-задач — лучший выбор. Работает под любой нагрузкой.

> **Почему не MySQL:** PostgreSQL лучше работает со сложными запросами, имеет больше возможностей для аналитики, не нагружает CPU на проценты.

### MySQL
**Для билетов:** 1, 5, 8, 18, 27

**Что сказать:**
> MySQL — самая популярная open-source СУБД. Быстрая, надёжная, огромный опыт эксплуатации. Для веб-проектов, блогов, интернет-магазинов — стандарт индустрии.

> **Почему не PostgreSQL:** MySQL проще в настройке, быстрее на типовых операциях чтения. Для простых CRUD-задач — идеально.

### SQLite
**Для билетов:** 1, 6, 8, 10, 27

**Что сказать:**
> SQLite — встраиваемая СУБД без сервера. Хранится в одном файле. Для прототипов, десктопных приложений, встроенных систем — идеально. Не требует установки и администрирования.

### MongoDB
**Для билетов:** 8

**Что сказать:**
> MongoDB — NoSQL СУБД, хранит данные в JSON-подобных документах. Для SPA-приложений, где схема данных часто меняется. Гибкая, горизонтально масштабируется.

### Redis
**Для билетов:** 8, 19, 24

**Что сказать:**
> Redis — in-memory хранилище данных. Используется как кэш, очередь сообщений (брокер), сессии. Молниеносный (микросекунды). Для онлайн-игр, чат-ботов, веба — необходим.

### InfluxDB
**Для билетов:** 25

**Что сказать:**
> InfluxDB — time-series СУБД для телеметрии, метрик, сенсоров. Оптимизирована для записи временных рядов. Для SCADA, энергетики, промышленной автоматизации — лучший выбор.

## 9. 🔒 БЕЗОПАСНОСТЬ

### auditd
**Для билетов:** 1, 4, 5, 13, 15, 16, 18

**Что сказать:**
> auditd — система аудита Linux. Отслеживает кто, когда и что делал в системе. Следит за /etc/passwd, изменениями конфигов, запусками программ. Для соответствия HIPAA/GDPR/PCI DSS — обязательно.

### ufw
**Для билетов:** 1, 4, 5, 13, 15, 18

**Что сказать:**
> ufw — межсетевой экран. Простая обёртка над iptables. "Закрыть все порты, открыть только нужные". Базовая защита сервера от внешних атак.

### WireGuard
**Для билетов:** 1, 13, 15

**Что сказать:**
> WireGuard — современный VPN. Простой, быстрый, безопасный (ChaCha20). Замена OpenVPN — в 10 раз быстрее, в 100 раз проще конфигурация.

### Fail2ban
**Для билетов:** 1, 13, 19

**Что сказать:**
> Fail2ban — защита от брутфорса. Следит за логами, банит IP после N неудачных попыток. Для SSH, веб-серверов — обязательно.

### ClamAV
**Для билетов:** 5, 6, 16, 18

**Что сказать:**
> ClamAV — бесплатный антивирус для Linux. Проверка файлов на вредоносное ПО. Особенно важно для билетов, связанных с финансами, медициной и безопасностью данных.

### SonarQube
**Для билетов:** 9, 13

**Что сказать:**
> SonarQube — статический анализ кода (SAST). Ищет уязвимости, code smells, дублирование. Безопасность на этапе разработки — devsecops.

### Trivy
**Для билетов:** 9, 13

**Что сказать:**
> Trivy — сканер уязвимостей контейнеров и зависимостей. Проверяет образы Docker, пакеты OS, библиотеки Python/JS. Быстрый, точный, бесплатный.

### nmap
**Для билетов:** 1, 4, 13, 15

**Что сказать:**
> nmap — сканер сети. Определяет открытые порты, запущенные сервисы, ОС. Для аудита сетевой безопасности — базовый инструмент.

### OpenSSL
**Для билетов:** 1, 10, 16, 18

**Что сказать:**
> OpenSSL — криптографическая библиотека. Генерация сертификатов, шифрование, TLS. Для HTTPS, защищённых соединений, ЭЦП.

## 10. 🛠️ УТИЛИТЫ

Все утилиты (Git, htop, curl/wget, vim/nano, p7zip, tree, net-tools, CUPS, Timeshift, build-essential, hardinfo, lm-sensors, screen/tmux, LibreOffice) — базовая инфраструктура, необходимая в любом билете.

### Git
**Что сказать:**
> Git — контроль версий. Откат изменений, ветвление, коллаборация. Без него — хаос. Обязателен в любой команде разработки.

### CUPS + cups-pdf
**Что сказать:**
> CUPS — система печати в Linux. cups-pdf — виртуальный принтер, создающий PDF из любого файла. Команда `lp -d PDF document.txt` → готовый PDF. Удобно для документации.

### Timeshift
**Что сказать:**
> Аналог "Восстановления системы" в Windows. Создаёт снэпшоты системы — если после установки ПО что-то сломалось, откатываемся за 1 минуту.

### htop
**Что сказать:**
> htop — диспетчер задач в терминале. Показывает загрузку CPU/RAM, запущенные процессы. Удобнее стандартного top.

## 📋 ШПАРГАЛКА ПО БАЛЛАМ (ЭТАП 2 — ОФИЦИАЛЬНЫЕ КРИТЕРИИ)

### Фраза для защиты "Стартовая настройка и интерфейс" (2.0 балла)

> "После установки ПО я выполнил стартовую настройку: настроил тему оформления, размер шрифта, автосохранение, включил необходимые расширения/плагины. Для VS Code установил расширения Python, GitLens, Docker — они нужны под задачи моего билета. Запустил приложение, убедился что интерфейс открывается и работает корректно. Git настроил: имя пользователя, email, ветка main по умолчанию."

### Фраза для защиты "Обмен данными" (2.0 балла)

> "Я настроил обмен данными между компонентами системы. Запустил и настроил СУБД (PostgreSQL/MySQL), создал тестовую базу данных. Инициализировал Git-репозиторий для контроля версий проекта. Создал каталоги для импорта и экспорта данных, настроил права доступа. Если требуется сетевое взаимодействие — Docker установлен и настроен."

### Фраза для защиты совместимости (2.50 балла)

> "Я настроил параметры совместимости системы для корректной работы установленного ПО. Настроил цветовую палитру и разрешение экрана, проверил отображение меню и кнопок, отключил композицию рабочего стола и масштабирование там, где это мешает работе приложений. Все пять параметров совместимости настроены."

Этап 2 даёт **9 баллов** из 40. Официальные критерии оценки:

| № | Что смотрят | Баллы | Что делать |
|---|---|---|---|
| 1 | **ПО выбрано с учетом задач организации** | **2.50** | **Расскажи по схеме** (см. выше): название → задачи → почему выбрал → альтернативы |

### Детали по настройке совместимости (5 × 0.50 = 2.50)

**4.1 Ограниченная цветовая палитра (0.50)**
- **Что сказать:** Настроил colord для управления цветовыми профилями, отключил ночной режим GNOME, чтобы не искажал цвета приложений. Для старых приложений через Wine настроил 16-битную палитру.

**4.2 Низкое разрешение (0.50)**
- **Что сказать:** Создал скрипт `run-lowres` — запускает приложения в 800×600. Для Wine — виртуальный рабочий стол 800×600. Это нужно для старых приложений, которые не поддерживают высокие разрешения.

**4.3 Отображение меню и кнопок (0.50)**
- **Что сказать:** Включил кнопки свернуть/развернуть/закрыть в заголовке окна, настроил тему Adwaita, установил qt5ct для Qt-приложений. Все элементы интерфейса отображаются корректно.

**4.4 Композиция рабочего стола (0.50)**
- **Что сказать:** Создал скрипт `disable-compositor` — отключает композицию GNOME для требовательных приложений (игры, 3D-редакторы). Это убирает задержки и артефакты, приложение работает напрямую с видеокартой.

**4.5 Масштабирование (0.50)**
- **Что сказать:** Выставил text-scaling-factor=1.0 и scaling-factor=0 — масштабирование отключено. Создал скрипт `disable-scaling` с переменными GDK_SCALE=1, QT_SCALE_FACTOR=1 для запуска конкретных приложений без масштаба.
